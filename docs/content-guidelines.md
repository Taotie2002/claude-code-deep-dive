# 内容粒度与表达规范

## 一、两类分析的深度标准

### 宏观分析（产品定位、设计哲学）
**适用场景**：第1-2章（产品定位）、设计哲学等章节

**分析深度标准**：
1. 明确核心观点（1-2句话）
2. 提供行业背景支撑（引用公开数据或行业报告）
3. 说明对开发者的实际意义
4. 避免陷入具体实现细节

**示例**：
```
✅ 正确：Claude Code定位为"AI结对编程伙伴"，强调人机协作而非替代
❌ 错误：Claude Code使用X技术栈实现Y功能模块
```

### 微观分析（数据流、工具调度、生命周期）
**适用场景**：Agent行为分析、数据流分析、错误处理等章节

**分析深度标准**：
1. 拆解核心流程（分步骤说明）
2. 标注关键节点和决策点
3. 提供逻辑验证或测试案例
4. 说明可能的边界条件和异常场景

**示例**：
```
✅ 正确：[L2] Agent接收用户指令后进入Planning状态，调用QueryEngine解析任务
         关键节点：任务分类 → 子任务分解 → 执行计划生成
❌ 错误：Claude Code的handleUserInput()函数在src/Agent/InputHandler.ts第45行
```

## 二、行文风格区分

### 主文档（技术教程）
**风格**：讲解式、步骤化、通俗解释

**特征**：
- 使用"我们来看"、"首先/然后"等引导语
- 配通俗类比帮助理解
- 每个概念后有小结或回顾
- 适合入门和系统学习

### docs/架构分析文档
**风格**：论证式、结论化、证据支撑

**特征**：
- 结论先行，再提供论据
- 标注[L1]/[L2]/[L3]/[L4]信源等级
- 提供可验证的测试案例
- 适合专业读者深度研究

## 三、章节结构模板

所有章节应遵循以下结构：

```markdown
## [章节标题]

### 核心目标
本章阐述XXX，帮助读者理解YYY。

### 内容概要
1. XXX（占比A%）
2. YYY（占比B%）

### 主体内容
...

### 关键结论
- 🔑 结论1：[信源等级]
- 🔑 结论2：[信源等级]

### 相关文档
- 相关章节：[链接]
- 进阶阅读：[链接]
```

## 四、术语使用规范

1. **首次定义**：每个术语首次出现时需标注类别
   - 🏛️ Anthropic官方术语
   - 🔧 行业通用术语
   - 🔮 项目推演术语

2. **持续使用**：首次定义后可简化标注

3. **避免混淆**：明确区分官方术语和推演术语
